## Why

当前系统将每个文件作为单一内容单元存储和向量化。对于长文档（如 8 万字 PDF），全文只生成一个向量，导致：
1. **语义被稀释**：单向量无法精确表达长文档中多个主题，搜索精度低
2. **无法定位具体位置**：搜索命中后用户只能看到整个文件，无法跳转到相关段落或视频时间点
3. **超长文本被截断**：嵌入模型通常有 token 限制（8192 token），超出部分丢失

需要引入语义分块机制，将长内容切分为语义连贯的小块，每块独立向量化，搜索时精准定位到具体段落/时间戳。

## What Changes

- 新增 `content_chunks` 表，存储分块文本、向量、位置信息（页码/偏移/时间戳）
- 引入语义切片服务，使用 BGE-M3 嵌入模型检测语义边界（本地/硅基流动免费 API）
- 修改内容处理管道：PDF 按语义分块 + 提取嵌入图片；音视频按字幕时间戳分块
- 修改搜索服务：从「整文档检索」升级为「语义块检索」，返回最相关块的精确位置
- 修改前端：搜索结果展示具体片段，点击可跳转到文档页码或视频时间点
- 新增 chunking 模型配置，与主嵌入模型（入库向量）独立配置

## Capabilities

### New Capabilities
- `content-chunking`: 内容语义分块——将长文档/音视频切分为语义连贯的小块，支持多种内容类型的差异化分块策略
- `chunk-search`: 分块级语义搜索——在语义块粒度执行向量检索，返回精确位置（文档页码/视频时间戳），支持前端跳转定位

### Modified Capabilities

## Impact

- **数据库**：新增 `content_chunks` 表（内容ID、分块文本、向量、位置元数据）
- **后端服务**：`process.py`（处理管道）、`embedding.py`（嵌入服务）、`search.py`（搜索服务）需重构
- **API**：搜索 API 响应增加分块位置信息（page_number、time_offset）；文件详情 API 返回分块列表
- **前端**：搜索结果页、内容详情页需支持位置跳转（PDF 页码定位、视频时间轴跳转）
- **配置**：`provider_configs` 新增 `chunking` 功能绑定，可独立配置语义切片模型
- **依赖**：新增 `semantic-text-splitter` 或 `sentence-transformers`（可选，用于本地语义切片）
